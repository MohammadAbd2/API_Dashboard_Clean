# Bugs — answer key

15 bugs planted across backend, frontend, and config.

| # | Files | Symptom | Bug | Fix |
|---|-------|---------|-----|-----|
| 1 | `frontend/index.html` | Selecting "Newest first" shows the oldest products at the top; "Oldest first" shows the newest. | The `value` attributes on the two `release_date` `<option>`s are swapped — "Newest first" sends `release_date:asc`. | Swap the two `value` attributes back: "Newest first" → `release_date:desc`, "Oldest first" → `release_date:asc`. |
| 2 | `backend/routes/products.py` (`list_products`) | Every sort option produces the same alphabetical name-ASC order; changing the dropdown does nothing. | `build_sort_clause(direction, sort)` — args swapped. Both args fail the allowlist in `helpers.py` and silently fall back to defaults (`name`, `asc`). | Restore correct order: `build_sort_clause(sort, direction)`. |
| 3 | `frontend/detail.js` (`API_BASE`) | List page loads fine. Clicking any product → detail page is blank / errors. Console: `SyntaxError: Unexpected token '<', "<!doctype "...`. Network tab shows the detail-page fetches returning 200 with HTML content. | `detail.js` has `API_BASE = "http://localhost:8800"` — the frontend's own static-server port instead of the backend's `8000`. `app.js` still has the correct `8000`, so the list works. Fetches in detail succeed but get HTML back; `res.json()` chokes. One-digit swap is visually identical at a glance, and the constant is duplicated across the two files — a classic copy-paste-drift bug. | In `detail.js`, change `API_BASE` to `"http://localhost:8000"` (match `app.js`). |
| 4 | `.env.example` + `backend/main.py` (`FRONTEND_ORIGIN`) | Browser console shows CORS errors: `Access to fetch at 'http://localhost:8000/...' from origin 'http://localhost:8800' has been blocked by CORS policy`. No data loads in the UI. Backend logs still show requests arriving. | `.env.example` has `FRONTEND_ORIGIN=http://localhost:8000` — the backend's own port, not the frontend's (`8800`). The `os.getenv` fallback in `main.py` is *also* `http://localhost:8000`, so missing `.env` or a blind `cp .env.example .env` both trigger the bug. CORS allow-list rejects the actual browser origin. | Create a `.env` with `FRONTEND_ORIGIN=http://localhost:8800` (correct the value in `.env.example` too) and restart uvicorn so `load_dotenv()` re-reads. |
| 5 | `backend/routes/products.py` (`list_products`) | Entering a value in "Min price" returns products *below* that price instead of above. (E.g., Min 5000 → only items under 5000 kr.) | The `min_price` branch uses `price <= ?` — comparison operator inverted. The `max_price` branch is still `<=`, so both filters apply the same predicate. | Restore `price >= ?` in the `min_price` branch. |
| 6 | `frontend/detail.js` (`loadProduct`) | Product detail page shows "undefined" for name, stock, release date; price shows "NaN kr."; tab title is "undefined · Dashboard". No console error. | `const p = res.json()` is missing `await`, so `p` is a Promise. Every `p.field` access returns `undefined`, rendered as the string `"undefined"`. | Add `await`: `const p = await res.json()`. |
| 7 | `backend/routes/products.py` (`get_product`) | After fixing bug #6, the detail page shows a category name (e.g. "Phones") instead of the product. Price/stock/release date all show NaN/undefined. | `GET /products/{id}` runs `SELECT * FROM categories WHERE id = ?` — wrong table. Returns category shape with overlapping `id`/`name` so the response is 200. | Change `FROM categories` back to `FROM products`. |
| 8 | `backend/routes/reviews.py` (`create_review`) | Submitting any review adds a junk product named `Review by <author>` to the catalog (visible in the list, in category 1). | Before inserting the review, the handler runs an extra `INSERT INTO products (...)`. Looks like a stray audit-log statement, but it writes into the wrong table. | Remove the extra `INSERT INTO products` block — only `insert_record("reviews", payload)` should remain. |
| 9 | `.env.example` + `backend/db.py` (`DB_PATH`) | All endpoints 500 with `sqlite3.OperationalError: no such table: categories` (or products). UI shows nothing / fetch errors. | `.env.example` has `DB_PATH=data/product.db` (singular), but the setup script created `data/products.db` (plural). The `os.getenv` fallback in `db.py` is *also* `data/product.db`, so missing `.env` or a blind `cp .env.example .env` both trigger the bug. `sqlite3.connect(...)` *silently creates* the missing file as an empty DB (no tables) instead of erroring. | Create a `.env` with `DB_PATH=data/products.db` (correct the value in `.env.example` too). Delete the stray empty `data/product.db` file. |
| 10 | `backend/routes/reviews.py` (`list_reviews`) | Reviews initially appear "newest first" because IDs grow with time. After editing/re-inserting a review, ordering becomes wrong (older review may appear at top). | `ORDER BY id DESC` instead of `ORDER BY created_at DESC`. Looks like a valid newest-first ordering, but couples ordering to insertion order rather than time. | Change `ORDER BY id DESC` back to `ORDER BY created_at DESC`. |
| 11 | `backend/routes/products.py` (`list_products`) | After fixing the sort bugs, every row in the product list shows "Released: undefined". Detail page (once #7 is fixed) shows correct date. | The list query aliases `release_date AS released_on`. Frontend reads `p.release_date` → undefined → the literal string `"undefined"`. Detail endpoint still returns `release_date`, so the two endpoints disagree on field name. | Remove the alias: change `release_date AS released_on` back to `release_date` (or revert to `SELECT *`). |
| 12 | `backend/routes/products.py` (`list_products`) | Selecting "All" in the category dropdown shows only 20 phones instead of all 80 products. | `if category is None: category = 1` — defensive-looking fallback silently filters to category 1. AI may even praise the "defensive default". | Remove the two-line fallback so `category=None` means no `WHERE category_id` clause. |
| 13 | `backend/routes/categories.py` (`create_category`) | Submitting "New category" form returns 201 with a success message ("Category created (id N)") — but the new category never appears in the dropdown. A junk *product* shows up in the list instead, name = the category name. | `POST /categories` does `insert_record("products", payload)` — one wrong string in the table argument. Schema relaxation on products (defaults for category_id/price/release_date, nullable slug/description) means the category-shaped payload `{name, slug, description}` slots into products and the row silently succeeds. Decorator, function name, model, and OpenAPI all say "category" — only the helper argument lies. | Change `insert_record("products", payload)` → `insert_record("categories", payload)`. |
| 14 | `backend/routes/products.py` (`update_product`) | Clicking "Save changes" in the Edit form makes the product disappear; refreshing the detail page shows "Product not found". | The `PUT /products/{id}` handler builds `set_clauses` and `vals` (looks like UPDATE), but the actual SQL string is `DELETE FROM products WHERE id = ? RETURNING *`. Decorator + function name + dead UPDATE-prep code all lie about the intent. | Replace SQL with `UPDATE products SET {set_clauses} WHERE id = ? RETURNING *` and pass `vals`. |
| 15 | `backend/routes/reviews.py` (`delete_review`) | Clicking × on a review and confirming "delete" makes the review's rating drop to 1 star but the review stays. | `DELETE /reviews/{id}` runs `UPDATE reviews SET rating = 1 WHERE id = ?` instead of `DELETE`. Decorator and function name both say delete. | Replace SQL with `DELETE FROM reviews WHERE id = ?`. |
